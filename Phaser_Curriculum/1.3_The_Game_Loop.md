# 1.3 — The Game Loop

## 大局观 (High-Level Map)

你打开一个游戏，角色在移动，敌人在攻击，画面在更新——这一切是怎么发生的？

答案是：**Game Loop（游戏循环）**。

游戏不是"响应式"的（不像网页，等用户点击才做事）。游戏是**主动的**：它每秒会执行几十次甚至几百次循环，每次循环都检查输入、更新状态、重新画面。只要游戏在运行，这个循环就不会停。

## 1. 为什么游戏需要循环？

想象你在玩一个平台跳跃游戏。你按住右方向键，角色就一直向右走。

这是怎么做到的？不是因为"按键触发了一个一次性的动作"，而是因为：

- **每一帧**，游戏都检查"右键现在是按下的吗？"
- 如果是，就把角色的位置向右移动一点点
- 然后把更新后的画面画出来
- 然后再重复

这就是 Game Loop——一个**持续运行、不断重复**的循环。

---

## 2. Game Loop 的三个阶段

每一次循环，游戏都经历三个阶段，顺序固定：

```
Input → Update → Render
```

### 2.1 Input（输入）

这一阶段，游戏**读取**玩家当前的输入状态：
- 哪些键盘按键被按下了？
- 鼠标在哪里？有没有点击？
- 手柄摇杆的值是多少？

注意：这里只是**读取**，不是处理。就像先把信息收集起来，稍后再用。

---

### 2.2 Update（更新）

这是游戏逻辑发生的地方。根据输入和游戏规则，更新所有游戏对象的**状态（state）**：

- 角色的位置、速度
- 敌人的 AI 行为
- 物理模拟（重力、碰撞）
- 分数、血量等数值

**State（状态）** 是指游戏中所有变量的当前值。Update 阶段就是根据规则修改这些值。

---

### 2.3 Render（渲染）

Update 结束后，所有对象的位置和状态已经更新好了。这一阶段负责把这些状态**画到屏幕上**：

- 根据每个对象的坐标，画出它的图像
- 清除上一帧的画面，画上新的画面

重要：**先 Update，再 Render**。这确保你看到的画面永远反映最新的游戏状态。

---

### 2.4 完整流程图

```
游戏开始
    ↓
┌─────────────────┐
│  Input          │  ← 读取玩家输入
│     ↓           │
│  Update         │  ← 更新游戏逻辑和状态
│     ↓           │
│  Render         │  ← 把新状态画到屏幕
└────────┬────────┘
         │ 循环，直到游戏结束
         └──────────────────→
```

## 3. Frame 和 Frame Rate

每一次完整的 Input → Update → Render 循环，叫做一 **Frame（帧）**。

**Frame Rate（帧率）** 就是每秒执行多少帧，单位是 **FPS（Frames Per Second，每秒帧数）**。

- 60 FPS = 每秒执行 60 次循环
- 30 FPS = 每秒执行 30 次循环

帧率越高，游戏越流畅。但帧率受到硬件性能限制——如果每帧的计算量太大，帧率就会下降。

**Frame-based Update** 的意思是：游戏的每一次逻辑更新，都是以"帧"为单位触发的。每帧执行一次 Update。

---

## 4. Delta Time

这是一个重要的概念。

**问题：** 如果你的游戏在 60 FPS 的电脑和 30 FPS 的电脑上运行，角色的移动速度会一样吗？

不会。如果你在 Update 里写"每帧移动 5 像素"：
- 60 FPS 的电脑：每秒移动 300 像素
- 30 FPS 的电脑：每秒移动 150 像素

**同一个游戏，在不同性能的设备上速度不一样** ——这是个 bug。

**Delta Time（时间差）** 就是解决这个问题的方法。

**Delta Time** 是指**上一帧到这一帧之间经过的时间**，单位通常是毫秒（ms）或秒（s）。

- 60 FPS 时，每帧约 16.7ms，delta time ≈ 0.0167s
- 30 FPS 时，每帧约 33.3ms，delta time ≈ 0.0333s

如果你把移动量乘以 delta time，比如"每秒移动 300 像素"：
- 60 FPS：每帧移动 300 × 0.0167 ≈ 5 像素
- 30 FPS：每帧移动 300 × 0.0333 ≈ 10 像素

两种情况，**每秒移动的总距离都是 300 像素**。这就实现了帧率无关的一致游戏体验。

> Phaser 在 `update(time, delta)` 函数里自动把 delta time 作为参数传给你。

## Self-Check

**Q1.** 什么是 Game Loop？为什么游戏需要它？

> **答：** Game Loop 是游戏持续执行的循环，每帧都会检查输入、更新状态、重新渲染画面。游戏需要它是因为游戏状态需要每帧主动更新，而不是被动等待用户触发。

---

**Q2.** Game Loop 的三个阶段是什么？它们的顺序是什么？

> **答：** Input → Update → Render。先读取输入，再根据输入更新游戏状态，最后把更新后的状态画到屏幕上。

---

**Q3.** 60 FPS 是什么意思？

> **答：** 60 FPS（Frames Per Second）表示每秒执行 60 次完整的 Game Loop 循环（Input → Update → Render），帧率越高游戏越流畅。

---

**Q4.** 一个游戏在 Update 阶段写的是"每帧移动 5 像素"。这个游戏在 60 FPS 和 30 FPS 的设备上，角色每秒移动的距离分别是多少？这说明了什么问题？

> **答：** 60 FPS 每秒移动 300 像素，30 FPS 每秒移动 150 像素。说明用"每帧"作为单位来控制速度，会导致游戏在不同性能设备上表现不一致。

---

**Q5.** Delta Time 是什么？它是如何解决帧率不一致问题的？

> **答：** Delta Time 是上一帧到这一帧之间经过的时间。把移动速度设为"每秒移动多少像素"并乘以 delta time，就能确保无论帧率高低，每秒实际移动的距离都相同，从而实现一致的游戏体验。
