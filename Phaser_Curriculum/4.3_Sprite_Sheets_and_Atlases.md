# 4.3 — Sprite Sheets and Atlases

## 大局观 (High-Level Map)

在 4.2 你学了 Sprite 是怎么显示**一张图片**的。但实际游戏里，角色不是静止不动的 — 他们会走路、跳跃、攻击，这需要**一系列图片**来组成动画。

问题是：如果每一帧都是一个独立的文件，加载 100 帧就需要 100 次网络请求，效率极低。

解决方案是 **Sprite Sheet** 和 **Texture Atlas** — 把多张图片**合并成一张大图**，一次请求搞定。

```
大图：
[ 多个独立图片文件 ] → [ 合并成一张大图 ] → [ Phaser 知道怎么切割它 ]
     (慢，多次请求)                              (快，一次请求)
```

这是理解 4.4 动画（Animations）的前提。

## 1. 为什么不用多张独立图片？

假设一个走路动画有 8 帧，每帧一个 `.png` 文件：

```
walk_01.png
walk_02.png
walk_03.png
...
walk_08.png
```

问题：
- 浏览器需要发 **8 次 HTTP 请求**，每次都有网络延迟
- 8 个文件分别存储，**磁盘空间浪费**（每个 PNG 文件都有自己的文件头信息）
- 管理麻烦，100 个动画可能就是 1000 个文件

**更好的做法是：把所有帧合并成一张大图。**

---

## 2. 什么是 Sprite Sheet（精灵表）？

**Sprite Sheet** 是把多张图片按照**固定格子（grid）**排列在一张大图里。

```
┌─────┬─────┬─────┬─────┐
│  0  │  1  │  2  │  3  │  ← 第一行，帧 0~3
├─────┼─────┼─────┼─────┤
│  4  │  5  │  6  │  7  │  ← 第二行，帧 4~7
└─────┴─────┴─────┴─────┘
```

**关键特点：**
- 每一格的大小**完全相同**（例如每格 64×64 px）
- 帧的顺序从左到右、从上到下
- Phaser 只需要知道"每格多宽多高"，就能自动切割

加载 Sprite Sheet 时需要告诉 Phaser 每格的尺寸：
```javascript
this.load.spritesheet('player', 'assets/player.png', {
    frameWidth: 64,
    frameHeight: 64
});
```

Phaser 会自动计算出有多少帧，并给它们编号（从 0 开始）。

## 3. 什么是 Texture Atlas（纹理图集）？

> ### 💡 "Atlas" 这个词是什么意思？
>
> **Atlas** 这个词来自希腊神话里的泰坦神 **Atlas（阿特拉斯）**，他被惩罚用双肩**扛着整个地球**。后来人们把"把很多东西集合在一起"的事物都叫 Atlas：
>
> - **地图册（Atlas）**：把世界上所有地图集合在一本书里，翻一本书就能查全世界。
> - **Texture Atlas（纹理图集）**：把游戏里所有贴图集合在一张大图里，加载一张图就拿到全部素材。
> - **MongoDB Atlas**：把数据库服务（存储、备份、监控、扩展）全部集合在一个云平台里，一个地方管理所有数据库需求。
>
> **核心含义始终一样：** 把很多分散的东西，**集中放在同一个地方**，方便统一管理和访问。

**Texture Atlas** 也是把多张图片合并成一张大图，但不同的是：**每张图片的大小可以不一样**，位置也是不规则的。

```
┌──────────────────────────────────────┐
│  ┌────┐  ┌──────────┐               │
│  │walk│  │  enemy_1 │  ┌──┐         │
│  │ _1 │  │          │  │bg│         │
│  └────┘  └──────────┘  └──┘         │
│       ┌─────┐  ┌──────────────┐     │
│       │item │  │   player_run │     │
│       └─────┘  └──────────────┘     │
└──────────────────────────────────────┘
```

因为大小不规则，Phaser 还需要一个额外的 **JSON 文件**来描述每张子图的位置和大小：

```json
{
  "walk_1": { "x": 10, "y": 10, "width": 64, "height": 64 },
  "enemy_1": { "x": 84, "y": 10, "width": 128, "height": 96 }
}
```

加载方式：
```javascript
this.load.atlas('characters', 'assets/characters.png', 'assets/characters.json');
```

### Sprite Sheet vs Texture Atlas 对比

| 特点 | Sprite Sheet | Texture Atlas |
|------|-------------|---------------|
| 每格大小 | **必须相同** | 可以不同 |
| 需要 JSON 文件 | 不需要 | **需要** |
| 空间利用率 | 较低（可能有空格） | **更高** |
| 使用场景 | 单个角色的动画帧 | 多种资源混合打包 |

## 4. Frame Index（帧索引）

**Frame Index** 是每一帧在 Sprite Sheet 里的编号，从 **0** 开始计数。

```
┌─────┬─────┬─────┬─────┐
│  0  │  1  │  2  │  3  │
├─────┼─────┼─────┼─────┤
│  4  │  5  │  6  │  7  │
└─────┴─────┴─────┴─────┘
```

你可以用 Frame Index 来指定 Sprite 显示哪一帧：
```javascript
this.add.sprite(100, 100, 'player', 3);
//                                   ↑ 显示第 3 帧（第一行最后一格）
```

在定义动画时（4.4 会学），你也是通过 Frame Index 来指定动画的帧序列：
```javascript
frames: [{ key: 'player', frame: 0 }, { key: 'player', frame: 1 }, ...]
// 或者更简洁地用范围
frames: this.anims.generateFrameNumbers('player', { start: 0, end: 3 })
```

对于 Texture Atlas，帧不是用数字索引，而是用 JSON 里的**名字（字符串）**来引用。

## 5. 内存效率 vs 批量渲染

使用 Sprite Sheet / Atlas 有两个核心好处：

### 内存效率（Memory Efficiency）

把 100 张 64×64 的小图合并成一张 800×800 的大图：
- 每个 PNG 文件都有文件头（header）和元数据，100 个文件 = 100 份开销
- 合并成一张图 = 只有 1 份文件头开销
- GPU 存储纹理也更高效，因为大块连续内存比碎片化的小块更好管理

### 批量渲染（Batched Rendering）

GPU 渲染图片的时候，**切换纹理（Texture Switch）是很贵的操作**。

如果场景里有 50 个 Sprite，各自用不同的图片文件：
- GPU 需要切换纹理 50 次
- 每次切换都会打断渲染流程（叫做 **draw call**）

如果 50 个 Sprite 都来自同一张 Atlas：
- GPU 整个场景只用**一张纹理**
- 可以把 50 个 Sprite 的绘制**打包成一次 draw call**
- 性能大幅提升

> **总结：** Sprite Sheet 和 Atlas 不只是"方便管理"的技巧，它们是性能优化的重要手段。

## Self-Check

**Q1.** Sprite Sheet 和 Texture Atlas 最根本的区别是什么？

> **答：** Sprite Sheet 要求每一帧的大小**完全相同**，按固定网格排列，不需要 JSON 文件。Texture Atlas 允许每张子图大小不同、位置不规则，但需要一个 JSON 文件来记录每张子图的位置和尺寸信息。

---

**Q2.** 一个 Sprite Sheet 有 3 行 4 列（每格 64×64），第二行第三格的 Frame Index 是多少？

> **答：** 帧从 0 开始，从左到右、从上到下编号。
> - 第一行：0, 1, 2, 3
> - 第二行：4, 5, **6**, 7
>
> 第二行第三格 = Frame Index **6**。

---

**Q3.** 为什么把多个 Sprite 放进同一张 Atlas 可以提升渲染性能？

> **答：** GPU 渲染时，切换纹理（Texture Switch）是昂贵的操作。如果所有 Sprite 都来自同一张 Atlas，GPU 只需要绑定一张纹理，就可以把所有 Sprite 的绘制操作**批量合并成一次 draw call**，大幅减少 GPU 的切换开销，提升渲染性能。
