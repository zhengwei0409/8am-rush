# 3.3 — Multi-Scene Architecture

## 大局观 (High-Level Map)

你现在知道 Scene 是什么，也知道每个 Scene 内部的生命周期方法怎么用。

这一节往上看一层：**当游戏有多个 Scene 时，它们是怎么组织的？**

真实游戏不会把所有东西塞进一个 Scene——那会变得混乱且难以维护。正确的做法是按职责把游戏拆分成多个独立的 Scene，每个 Scene 只负责一件事。这就是 **Multi-Scene Architecture（多场景架构）**。

---

1. 为什么需要多个 Scene？

## 1. 为什么需要多个 Scene？

想象把一个游戏的所有逻辑都写在一个 Scene 里：
- 加载画面的逻辑
- 主菜单的按钮逻辑
- 游戏进行中的角色移动、敌人 AI、碰撞
- 游戏结算的分数显示

这些逻辑会互相混在一起，代码很快变得难以阅读和维护。

**多场景架构的核心思想：** 按"职责"把游戏拆分。每个 Scene 只做一件事，Scene 之间通过 Scene Manager 来协调。

---

## 2. 四种常见的 Scene 角色

### 2.1 Boot Scene — 启动场景

**职责：** 游戏最先运行的 Scene，负责加载游戏运行所需的最基础资源（比如加载画面本身用到的图片），然后立刻跳转到下一个 Scene。

Boot Scene 通常非常简单，执行时间极短。玩家甚至感觉不到它的存在。

```
游戏启动 → Boot Scene（加载最基础资源）→ 跳转到 Preload Scene 或 Menu Scene
```

---

### 2.2 Menu Scene — 主菜单场景

**职责：** 显示游戏的主菜单界面，处理玩家在菜单里的操作（开始游戏、查看设置、退出等）。

Menu Scene 是玩家进入游戏后最先**看到**的画面，通常包含：
- 游戏标题
- 开始按钮
- 设置选项

---

### 2.3 Gameplay Scene — 游戏主场景

**职责：** 这是游戏的核心 Scene，包含所有游戏玩法逻辑：
- 角色移动和控制
- 敌人 AI
- 物理和碰撞
- 关卡结构

Gameplay Scene 通常是整个项目里最复杂的 Scene，但因为它和其他 Scene 是隔离的，复杂性被控制在这一个 Scene 内部。

---

### 2.4 UI Overlay Scene — UI 叠加场景

这是一个稍微高级一点的概念，但值得了解。

**UI（User Interface，用户界面）** 指的是玩家在游戏中看到的信息显示，比如：血条、分数、弹药数量、小地图。

**Overlay（叠加）** 的意思是：这个 Scene 不是独立显示的，而是**叠加在 Gameplay Scene 之上**。

Phaser 允许多个 Scene **同时运行并叠加显示**：

```
┌─────────────────────────┐
│   UI Overlay Scene      │  ← 上层：血条、分数（始终显示）
│  ❤️❤️❤️  Score: 1200   │
├─────────────────────────┤
│   Gameplay Scene        │  ← 下层：游戏世界、角色、敌人
│   🧑 ← 🟩🟩🟩🟩       │
└─────────────────────────┘
```

把 UI 单独放在一个 Scene 的好处是：Gameplay Scene 可以在不影响 UI 的情况下重新开始（比如玩家死亡后重置关卡），UI 保持稳定运行。

---

## 3. Scene Manager — 场景管理器

**Scene Manager** 是 Phaser 内置的系统，负责管理游戏里的所有 Scene。

你在 Config 的 `scene` 数组里注册所有 Scene 之后，Scene Manager 就掌管了它们：

| Scene Manager 的能力 | 说明 |
|----------------------|------|
| 启动 Scene | 让一个 Scene 开始运行 |
| 停止 Scene | 让一个 Scene 停止运行（保留状态） |
| 暂停 Scene | 暂停 update()，但 Scene 还在 |
| 切换 Scene | 停止当前 Scene，启动另一个 |
| 同时运行多个 Scene | 用于 UI Overlay 这类需求 |

在代码里，你通过 `this.scene` 来访问 Scene Manager，例如：

```javascript
// 在 GameScene 里，游戏结束时切换到 GameOver Scene
this.scene.start("GameOver");
```

---

## 4. 场景分离的好处

**Scene Separation（场景分离）** 是 Multi-Scene Architecture 的核心实践。

| 好处 | 说明 |
|------|------|
| **职责清晰** | 每个 Scene 只做一件事，代码逻辑容易找 |
| **独立开发** | 团队不同成员可以同时开发不同 Scene |
| **独立测试** | 可以直接启动某个 Scene 来测试，不需要从头玩 |
| **避免状态污染** | Scene 之间状态隔离，一个 Scene 的 bug 不会影响另一个 |

## Self-Check

**Q1.** 为什么不把所有游戏逻辑写在一个 Scene 里？

> **答：** 把所有逻辑放在一个 Scene 里会导致代码混杂在一起，难以阅读和维护。多场景架构按职责拆分，每个 Scene 只负责一件事，结构更清晰。

---

**Q2.** Boot Scene 的职责是什么？它和 Menu Scene 的区别是？

> **答：** Boot Scene 是游戏最先运行的场景，负责加载最基础的资源后立刻跳转，玩家几乎感觉不到它。Menu Scene 是玩家实际看到的主菜单画面，提供开始游戏等操作入口。

---

**Q3.** 什么是 UI Overlay Scene？为什么要把 UI 单独放在一个 Scene 里？

> **答：** UI Overlay Scene 是一个叠加显示在 Gameplay Scene 上方的场景，用来显示血条、分数等信息。单独放的好处是：Gameplay Scene 可以重置（比如玩家死后重开），UI Scene 继续稳定运行，两者互不干扰。

---

**Q4.** Scene Manager 的作用是什么？开发者怎么使用它？

> **答：** Scene Manager 是 Phaser 内置的系统，负责管理所有已注册 Scene 的启动、停止、切换和叠加运行。开发者通过 `this.scene` 在代码里访问它，例如 `this.scene.start("GameOver")` 来切换场景。

---

**Q5.** Multi-Scene Architecture 对团队开发有什么好处？

> **答：** 每个 Scene 职责独立，团队成员可以同时开发不同的 Scene 而不互相干扰。也可以单独测试某个 Scene，而不需要从游戏开头走一遍。
