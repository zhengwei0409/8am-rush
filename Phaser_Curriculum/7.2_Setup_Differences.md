# 7.2 — Setup Differences

## 大局观 (High-Level Map)

在 7.1 你了解了 Phaser 4 的定位：TypeScript-first、模块化、支持 Tree-shaking。

这些设计决定直接影响了**项目怎么搭建**。Phaser 3 和 Phaser 4 的 Setup（初始化过程）有根本性的差异。

这节的目标：理解为什么 Setup 会不同，以及不同在哪里。

```
大图：
Phaser 3 Setup → 一个 HTML 文件 + 一个 script 标签搞定
Phaser 4 Setup → 需要打包工具 + ES Module imports
```

## 1. Phaser 3 的 Setup 方式回顾

Phaser 3 的最简 Setup：

```html
<!-- index.html -->
<!DOCTYPE html>
<html>
<body>
  <script src="phaser.min.js"></script>  <!-- 整个引擎 -->
  <script src="game.js"></script>        <!-- 你的游戏代码 -->
</body>
</html>
```

```javascript
// game.js
const config = { ... };
const game = new Phaser.Game(config);  // Phaser 是全局变量
```

特点：
- 不需要任何构建工具
- 直接在浏览器打开 `index.html` 就能运行（配合本地 server）
- `Phaser` 是一个**全局对象**，整个引擎挂在 `window.Phaser` 上
- 引入一个文件，获得所有功能

这种方式简单，但文件体积大（整个 Phaser 3 约 1MB+），无法做 Tree-shaking。

---

## 2. ES Module Imports 是什么？

**ES Module** 是 JavaScript 官方的模块系统，使用 `import` 和 `export` 语法。

```javascript
// ES Module 语法
import { Game, Scene } from 'phaser';
import { ArcadePhysics } from 'phaser/physics';
```

ES Module 的核心思想：**每个文件明确声明它需要什么，提供什么**，而不是靠全局变量共享。

Phaser 3 的传统方式是通过 `<script>` 标签把 Phaser 挂到全局，这是 ES Module 之前的老做法。

Phaser 4 完全基于 ES Module，你通过 `import` 语句引入你需要的功能，而不是加载一个全局脚本。

---

## 3. Per-feature Imports（按需引入）

Phaser 4 的模块化让你只引入游戏真正需要的部分：

```javascript
// Phaser 3：只能引入整个引擎
import Phaser from 'phaser';  // 全部功能都加载了

// Phaser 4：按需引入
import { Game } from 'phaser/core';
import { Scene } from 'phaser/scene';
import { Sprite } from 'phaser/gameobjects/sprite';
// 不用 Physics？就不 import，打包里就没有 Physics 的代码
```

这是 Tree-shaking 能发挥作用的前提：打包工具分析你的 `import`，只打包你实际引用到的模块。

**对开发体验的影响：**
- 你需要知道每个功能在哪个模块里
- 但编辑器会帮你自动补全，实际不麻烦
- 换来的是更小的最终文件

---

## 4. 为什么 Phaser 4 需要 Bundler？

**Bundler（打包工具）** 是一个把你的源码（多个 `.js`/`.ts` 文件 + `node_modules` 里的库）打包成浏览器能直接运行的一个或多个文件的工具。常见的有 **Vite**、**Webpack**、**Rollup**。

**Phaser 3 不需要 Bundler（可以直接用 `<script>` 标签）**，因为它是一个单独的 `.js` 文件，直接在浏览器里运行。

**Phaser 4 需要 Bundler**，原因：

1. Phaser 4 以 NPM 包的形式分发，分散在很多小模块里，浏览器不能直接处理几百个独立的 `.js` 文件
2. TypeScript 源码需要先编译成 JavaScript，Bundler 负责这步
3. Tree-shaking 是 Bundler 执行的，没有 Bundler 就没有 Tree-shaking

```
你的源码（TypeScript）
       ↓
    Bundler（Vite / Webpack）
       ↓
最终打包文件（浏览器能运行的 .js）
```

Phaser 4 项目的标准工作流：
```
npm install → 写代码（TypeScript）→ 打包工具处理 → 浏览器运行
```

---

## 5. Smaller Bundle Size（更小的打包体积）

Per-feature Imports + Bundler + Tree-shaking 的组合效果：

**Phaser 3：**
- 引入整个引擎：~1MB+
- 你只用了 10% 的功能，但 100% 的代码都打包进去了

**Phaser 4（只用 Arcade Physics + Sprite + Scene）：**
- 只打包你用到的模块：可能只有 200~300KB
- 用户下载的文件更小，游戏加载更快

对于手机网络或弱网环境，文件体积直接影响玩家等待时间，是实际的用户体验指标。

---

## 6. No Global Phaser Object（没有全局 Phaser 对象）

Phaser 3 里，你可以在代码任何地方用 `Phaser.xxx`：

```javascript
// Phaser 3：Phaser 是全局变量，随处可用
const renderer = Phaser.AUTO;
const key = Phaser.Input.Keyboard.KeyCodes.SPACE;
```

Phaser 4 里，**没有全局的 `Phaser` 对象**。你要用什么，就在文件顶部 `import` 什么：

```javascript
// Phaser 4：必须明确 import
import { AUTO } from 'phaser/const';
import { KeyCodes } from 'phaser/input/keyboard';

const renderer = AUTO;
const key = KeyCodes.SPACE;
```

这是 ES Module 的标准做法 — 依赖关系明确、可追踪，不依赖全局变量的隐式共享。

**好处：**
- 每个文件的依赖关系一目了然（看 import 就知道这个文件用了什么）
- 避免命名冲突（不同库的全局变量可能同名）
- Bundler 能正确分析依赖，执行 Tree-shaking

---

## Self-Check

**Q1.** Phaser 4 为什么需要 Bundler，而 Phaser 3 不需要？

> **答：** Phaser 3 是一个单独的 `.js` 文件，可以直接用 `<script>` 标签引入，浏览器能直接运行。Phaser 4 以 NPM 包形式分发，源码是 TypeScript，拆分成很多小模块，浏览器无法直接处理。Bundler 负责把 TypeScript 编译成 JavaScript、把所有模块合并、执行 Tree-shaking，最终生成浏览器能运行的文件。

---

**Q2.** Per-feature Imports 和 Tree-shaking 怎么配合，达到减小打包体积的效果？

> **答：** Per-feature Imports 让你只 `import` 游戏用到的功能模块。Bundler 在打包时分析这些 `import` 语句，发现没有被 `import` 的模块，Tree-shaking 把它们从最终文件中移除。两者配合的结果是：打包文件只包含游戏实际使用的代码，未使用的功能（如你没用到的 Sound 系统、Matter.js 等）完全不出现在最终文件里。

---

**Q3.** Phaser 3 用 `Phaser.Input.Keyboard.KeyCodes.SPACE` 访问键码，Phaser 4 里要怎么做到同样的事？

> **答：** Phaser 4 没有全局 `Phaser` 对象，需要在文件顶部明确 import：
> ```javascript
> import { KeyCodes } from 'phaser/input/keyboard';
> // 然后使用
> const spaceKey = KeyCodes.SPACE;
> ```
> 依赖关系通过 `import` 显式声明，而不是靠全局变量。
